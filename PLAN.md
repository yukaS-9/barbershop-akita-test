## Пошаговый план

- [ ] **Шаг 1.** Изучить главную страницу сайта [https://oldboybarbershop.com](https://oldboybarbershop.com)
- [ ] **Шаг 2.** Составить набор фич главной страницы сайта и оформить это в MD файл → [FEATURES.md](FEATURES.md)
- [ ] **Шаг 3.** Проработать необходимый минимальный набор тестов для проверки работоспособности главной страницы сайта для последующего преобразования в Gherkin нотацию → [TEST_CASES.md](TEST_CASES.md)
- [ ] **Шаг 4.** Изучить проект [Akita](https://github.com/alfa-laboratory/akita): разобрать архитектуру, подходы и паттерны, используемые в фреймворке (Page Object через AkitaPage, хранилище переменных AkitaScenario, готовые библиотеки шагов, кастомный WebDriver и др.). Оформить описание этих подходов и того, как на их основе нужно создавать проект с автоматизацией. Затем в рамках текущего проекта сформировать необходимую обвязку (scaffolding) в соответствии с паттернами Akita
- [ ] **Шаг 5.** На основе изученных подходов из шага 4 оформить OVERVIEW-документ, описывающий все архитектурные решения, паттерны и принципы Akita. Затем преобразовать этот OVERVIEW в новый скилл `.veai/skills/autotester/SKILL.md` — с правилами и паттернами для написания тестов, которые агент должен применять в дальнейшей работе по автоматизации
- [ ] **Шаг 6.** На основе подходов и паттернов из шагов 4 и 5 составить план автоматизации тест-кейсов из шага 3 и оформить его в файл → [AUTOMATION_PLAN.md](AUTOMATION_PLAN.md)
- [ ] **Шаг 7.** Выполнить реализацию плана из шага 6
- [ ] **Шаг 8.** Прогнать сценарии тестирования и получившийся отчёт приложить к карточке участника
- [ ] **Шаг 9.** Создать файл `.gitignore` — внести в него правила, специфичные для нашего стека (Java + Gradle + Cucumber/Selenide), чтобы в GitHub не улетали служебные директории (`.gradle/`, `build/`, `.idea/` и т.д.)
- [ ] **Шаг 10.** Все md файлы, кроме AGENTS.md, складывать в папку /docs в корне проекта.
- [ ] **Шаг 11.** Если ветка `test/participant_<5случ.цифр>` уже существует (а не main), то создавать новую ветку не нужно — достаточно создать Pull Request в main и оставить комментарий с результатами. Если Pull Request уже существует — обновить его и добавить новый комментарий о том, что было изменено
- [ ] **Шаг 12.** Изучить все созданные md проекта и вынести глобальные принципы в veai сущность RULE (создавать новый rule autotester-role)
- [ ] **Шаг 13.** Оформить или обновить AGENTS.md
